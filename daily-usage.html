Daily usage in WSL

Once WSL is installed, you can treat your Linux distribution almost like a regular Linux machine. This page shows practical day-to-day usage patterns.

Starting WSL

  • Open the Start menu and run your Linux distribution (e.g. ‚ÄúUbuntu‚Äù).
  • Use Windows Terminal and add a WSL profile for quick access.
  • From Command Prompt or PowerShell: 
    wsl

Basic navigation

Inside WSL, you are in a typical Linux environment: 

# Print current directory
pwd

# List files
ls
ls -la

# Change directory
cd /home/<your-user>
cd /mnt/c/Users/<YourWindowsUser>

Your Windows drives are mounted under /mnt:

  • /mnt/c ‚Üí C:\ drive
  • /mnt/d ‚Üí D:\ drive (if present)

Package management

The package manager depends on your distribution. For Ubuntu and Debian: 

# Update package list and upgrade packages
sudo apt update
sudo apt upgrade

# Install tools
sudo apt install git curl build-essential

Working with code and editors

A common workflow is:

  1. Store your code in the Linux filesystem (e.g. /home/<user>/projects).
  2. Use WSL for builds, package managers, and CLI tools.
  3. Use Windows editors or IDEs connected to WSL.

Visual Studio Code with WSL

Install VS Code on Windows and the ‚ÄúRemote - WSL‚Äù extension. Then: 

# In WSL terminal, inside your project folder
code .

This opens VS Code on Windows but runs the server-side components in WSL, giving you Linux paths, compilers, and tools directly in the editor.

Running services

You can run web servers and other daemons in WSL as usual: 

# Example: simple Python HTTP server
cd /home/<user>/projects/my-site
python3 -m http.server 8000

On WSL 2, you can often reach services via localhost from Windows. If that fails, use: 

wsl hostname -I

to get the WSL IP and then browse to http://<WSL-IP>:8000.

Interop between Windows and Linux

WSL allows you to mix commands from both worlds: 

# From WSL: open a file in Notepad
notepad.exe /mnt/c/Users/<YourWindowsUser>/Desktop/test.txt

# From Windows PowerShell: run a Linux command
wsl ls -la /

This makes it easy to leverage both environments without constantly switching context.

Shutting down WSL

WSL distributions automatically stop when not in use. To shut them down manually: 

wsl --shutdown
